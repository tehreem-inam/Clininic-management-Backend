# Clinic Management Backend

A FastAPI backend for clinic management with SQLAlchemy and Alembic migrations.

## Overview
- App layout: code lives under `app/`.
- Migrations: `app/migrations` (Alembic script_location).
- DB: PostgreSQL (async runtime uses `asyncpg`, migrations use sync driver substitution).

## Prerequisites
- Python 3.13 (use a virtualenv at `.venv`).
- PostgreSQL server available and reachable from `DATABASE_URL`.

## Quick setup
1. Create and activate virtualenv:

    python -m venv .venv
    source .venv/bin/activate
    pip install -r requirements.txt

2. Create a `.env` file in the project root with at least:

    DATABASE_URL=postgresql+asyncpg://user:pass@localhost:5432/dbname
    # other secrets (do not commit .env)

3. Makefile and commands below auto-load `.env` before running.

## Makefile targets (common)
- `make alembic-rev m="message"` — create an Alembic revision with autogenerate.
- `make alembic-upgrade` — run `alembic -c alembic.ini upgrade head`.
- `make alembic-stamp` — stamp the current revision without applying SQL (use when DB already has schema).
- `make local-db-upgrade` — loads `.env` then runs upgrade against local DB (convenience alias).
- `make run` — start development server (uvicorn with reload).
- `make shell` — open a shell with `.env` loaded.

Examples:

    # create a new migration
    make alembic-rev m="add-prescription-fields"

    # apply migrations to local DB
    make local-db-upgrade

    # mark revision as applied without changing DB
    make alembic-stamp

    # run dev server
    make run

## Alembic notes
- Migrations live under `app/migrations`.
- `app/migrations/env.py` reads `DATABASE_URL` from `.env` and rewrites the URL for Alembic to use a sync driver (psycopg) when necessary. This avoids async-driver issues during migrations.
- If an autogenerated revision references `postgresql.JSONB` or other dialect types, Alembic will include the dialect import; if you see a NameError, ensure `from sqlalchemy.dialects import postgresql` is present in the revision file.
- If your DB already contains tables that conflict with new prefixed tables, either inspect and edit the generated revision or use `make alembic-stamp` to mark the migration as applied.

## Security & secrets
- `.env` should never be committed. The repo's `.gitignore` contains `.env` and `.venv`.
- If credentials were accidentally committed, rotate the secrets and consider rewriting git history (e.g., `git filter-repo`) — coordinate with your team.

## Troubleshooting
- Error: `DependentObjectsStillExist` when dropping tables — migrations that remove legacy tables must either drop with `CASCADE` or be adjusted to avoid destructive changes. Inspect generated revision before running `make local-db-upgrade` if you have existing data.
- Error: greenlet/async-driver when running Alembic — `env.py` handles `+asyncpg` -> `+psycopg` conversion; ensure `DATABASE_URL` set correctly in `.env`.
